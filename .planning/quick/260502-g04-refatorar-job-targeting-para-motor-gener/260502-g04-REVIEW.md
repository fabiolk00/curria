## Critical Issues

### CR-01: Accepted low-fit override can downgrade structured forbidden/cautious claim failures

**File:** `src/lib/agent/job-targeting-pipeline.ts:466`

**Issue:** Structured claim-policy failures are produced as high-severity issues in `validation-policy.ts`, including `forbidden_term` mapped to `forbidden_claim` and `unsafe_direct_claim` mapped to `unsupported_claim`. The accepted-low-fit path treats both issue types as audit-only: `ACCEPTED_LOW_FIT_AUDIT_ONLY_ISSUE_TYPES` includes `forbidden_claim` and `unsupported_claim`, `shouldBlockAfterAcceptedOverride()` only blocks issue types outside that allowlist, and the accepted override path can then call `relaxValidationForAcceptedLowFitOverride()` to clear `blocked`. That means a forbidden or cautious claim from `JobCompatibilityAssessment.claimPolicy` can be accepted away and persisted/generated after user override, which violates the structured validation safety invariant.

**Fix:** Keep assessment claim-policy violations non-overridable. Preserve and check the structured `issue.code` values from `buildAssessmentClaimPolicyIssues()` before relaxing validation, and only relax synthetic low-fit or explicitly designated recoverable low-fit issues.

```ts
const NON_OVERRIDABLE_STRUCTURED_CODES = new Set([
  'forbidden_term',
  'unsafe_direct_claim',
  'unsupported_skill_added',
  'unsupported_certification',
  'unsupported_education_claim',
  'target_role_asserted_without_permission',
])

export function shouldBlockAfterAcceptedOverride(validation: RewriteValidationResult): boolean {
  return validation.hardIssues.some((issue) =>
    !issue.issueType
    || !ACCEPTED_LOW_FIT_AUDIT_ONLY_ISSUE_TYPES.has(issue.issueType)
    || (issue.code !== undefined && NON_OVERRIDABLE_STRUCTURED_CODES.has(issue.code))
  )
}
```

Also add a regression test where a low-fit override is accepted but a generated CV still contains a `forbidden_term` or `unsafe_direct_claim`; the expected result should remain blocked and should not generate the artifact.

## Warnings

### WR-01: Assessment target-role extraction is weaker than the now-suppressed legacy extractor

**File:** `src/lib/agent/job-targeting/compatibility/assessment.ts:140`

**Issue:** `evaluateJobCompatibility()` only extracts a target role from explicit labels like `role:`, `title:`, `cargo:`, or `vaga:` and otherwise returns `Vaga Alvo` with low confidence. Because `buildBaseTargetingPlan()` now uses `params.jobCompatibilityAssessment` directly when present, it skips the older heuristic/LLM extraction path. Common job descriptions that put the title in the first line or fixture `input.job.title` but not in a labeled line will now lose the real role downstream in plans, modals, explanations, and comparison responses.

**Fix:** Make the assessment contract own a robust role field before it becomes the source of truth. Either move the legacy heuristic/LLM role extractor into the assessment path, or pass a precomputed target role into `evaluateJobCompatibility()` and audit its source there. Add tests with an unlabeled first-line title and use golden fixture `input.job.title` in assessment coverage.

### WR-02: Assessment-derived targeting plans still preserve legacy compatibility signals

**File:** `src/lib/agent/job-targeting/compatibility/legacy-adapters.ts:611`

**Issue:** When an assessment exists, `buildTargetedRewritePlan()` builds a legacy base plan and then calls `buildTargetingPlanFromAssessment()`. The adapter still prefers `basePlan.focusKeywords`, `basePlan.mustEmphasize`, and `basePlan.missingButCannotInvent` over assessment-derived requirements/gaps. These fields feed rewrite prompts and retry instructions, so a new Job Targeting run can still emphasize or suppress based on divergent legacy gap/keyword calculations even while score, low-fit, and validation use the assessment.

**Fix:** In assessment-backed runs, derive compatibility-bearing fields from the assessment by default. Keep `basePlan` only for non-compatibility presentation defaults if needed.

```ts
focusKeywords: options.focusKeywords
  ?? dedupe(assessment.requirements.flatMap((requirement) => requirement.extractedSignals)).slice(0, 16),
mustEmphasize: options.mustEmphasize
  ?? adapterParts.safeTargetingEmphasis.safeDirectEmphasis,
missingButCannotInvent: options.missingButCannotInvent
  ?? dedupe(assessment.criticalGaps.map((gap) => displaySignal(gap.signal))),
```

Update `legacy-adapters.test.ts`; the current test locks preservation of legacy emphasis even when an assessment is present.

### WR-03: Runtime still hardcodes catalog segments and concrete tooling examples

**File:** `src/lib/agent/job-targeting/compatibility/assessment.ts:35`

**Issue:** The assessment runtime hardcodes the default domain pack list by segment (`data-bi`, `software-engineering`, `finance`, etc.). Separately, `target-recommendations.ts:191` classifies tooling recommendations with concrete stack terms such as `sql`, `python`, `excel`, and `power bi`. This leaves part of the runtime outside the catalog source of truth: adding a new pack under `catalog/domain-packs/` will not affect assessments unless the hardcoded list is edited, and fallback recommendations can still behave differently from catalog-driven compatibility.

**Fix:** Centralize default domain pack discovery in the catalog loader and have assessment use that catalog source. Replace concrete tooling regexes in runtime recommendation fallback with catalog category/requirement-kind metadata, or expand the hardcode guard to cover any runtime fallback that is still considered part of core Job Targeting.
