# Milestone v1.4: Agent Core Modularization, Security Hardening, and Release Stability

**Status:** SHIPPED 2026-04-15
**Phases:** 28-31.1
**Total Plans:** 13

## Overview

`v1.4` hardened the brownfield core funnel in three directions: smaller agent-runtime seams, stronger authenticated and billing trust boundaries, and more reliable generation and release proof. The milestone also absorbed an urgent inserted phase, `31.1`, to reduce non-E2E runtime waste and expose repeatable performance proof in CI.

This milestone shipped with accepted audit debt: the implementation and targeted tests are present, but the formal phase `VERIFICATION.md` layer was not backfilled before archive.

## Phases

### Phase 28: Agent Input and Setup Service Extraction

**Goal**: Pull the front half of the agent runtime into smaller services so message preparation, vacancy detection, and pre-loop setup are easier to reason about and test.
**Depends on**: Nothing (first phase of milestone v1.4)
**Plans**: 3 plans

Plans:

- [x] 28-01: Extract message preparation into a dedicated service with route-level contract preservation
- [x] 28-02: Isolate vacancy detection and pre-loop setup behind explicit runtime boundaries
- [x] 28-03: Add targeted regression tests for extracted agent input and setup flow

**Details:**
- extracted `message-preparation`, `vacancy-analysis`, and `pre-loop-setup` into explicit services
- made the route-to-loop handoff explicit through `AgentLoopParams`
- added direct regression proof around vacancy detection, setup flow, and transformed route input

### Phase 29: Agent Recovery, Streaming, and Persistence Decomposition

**Goal**: Split the back half of the runtime into clearer retry, recovery, streaming, and persistence services with durable tests around handoff behavior.
**Depends on**: Phase 28
**Plans**: 3 plans

Plans:

- [x] 29-01: Extract retry and recovery policy into dedicated runtime services
- [x] 29-02: Separate streaming and persistence responsibilities from the central loop orchestration
- [x] 29-03: Add regression proof for recovery, streaming, and persistence handoffs

**Details:**
- extracted `agent-recovery`, `agent-streaming`, and `agent-persistence` from the oversized loop
- kept `agent-loop.ts` focused on orchestration instead of transport and persistence details
- closed the seams with direct tests plus reruns of route and streaming regressions

### Phase 30: Authenticated Route and Billing Boundary Hardening

**Goal**: Remove weak trust assumptions from checkout, redirect, callback, and mutation flows so sensitive routes fail closed and leave useful evidence.
**Depends on**: Phase 28
**Plans**: 2 plans

Plans:

- [x] 30-01: Harden canonical-host and return-path validation across checkout and external redirect flows
- [x] 30-02: Add explicit origin or CSRF enforcement and regression proof for sensitive authenticated mutations and billing callbacks

**Details:**
- replaced raw-origin checkout trust with canonical app-url derivation
- added shared request-trust validation for high-value authenticated mutations
- preserved explicit server-to-server trust on the Asaas webhook while rejecting hostile browser origins

### Phase 31: Long Vacancy Stability and Release Hygiene Gates

**Goal**: Eliminate the known generation instability and encoding defects, then raise CI gates around workspace, preview, and release-critical regressions.
**Depends on**: Phase 29, Phase 30
**Plans**: 2 plans

Plans:

- [x] 31-01: Fix long vacancy generation and encoding regressions with focused browser or route-level proof
- [x] 31-02: Strengthen CI and release gates for workspace, preview, and generation-state stability

**Details:**
- stabilized repeated long-vacancy generation in browser proof by removing optional billing drag and detached-frame reload behavior
- cleaned remaining mojibake expectations from release-facing regression coverage
- promoted core-funnel plus long-vacancy stability into an explicit release-critical E2E gate

### Phase 31.1: Reduce Test Suite Runtime and Add CI-Friendly Performance Proof (INSERTED)

**Goal**: Reduce the non-E2E test suite runtime to a stable CI-friendly budget, remove artificial test overhead, and leave measurable proof so regressions are visible before release.
**Depends on**: Phase 31
**Plans**: 3 plans

Plans:

- [x] 31.1-01: Lock the non-E2E runtime baseline and test-environment budget path
- [x] 31.1-02: Remove artificial delay from the slowest profiled non-E2E suites without weakening coverage
- [x] 31.1-03: Add repeatable runtime proof and closure criteria for remaining outliers

**Details:**
- formalized a node-first Vitest environment split and removed retry backoff sleeps from tests

## Milestone Summary

**Decimal Phases:**

- Phase 31.1: Reduce Test Suite Runtime and Add CI-Friendly Performance Proof (inserted after Phase 31 for urgent CI-feedback remediation)

**Key Decisions:**

- Continue the agent-runtime refactor as incremental service extraction instead of a broad rewrite.
- Move sensitive route trust to canonical-host and shared browser-trust contracts rather than raw request metadata.
- Treat long-vacancy stability and preview integrity as release-blocking quality concerns, not optional cleanup.
- Accept `v1.4` archive with explicit audit debt after generating a milestone audit that documented missing `VERIFICATION.md` artifacts.

**Issues Resolved:**

- oversized route and loop ownership in the core agent runtime
- weak checkout-origin trust and missing browser-trust enforcement on authenticated mutations
- repeated long-vacancy instability and remaining release-facing mojibake expectations
- structural non-E2E test waste from global `jsdom` and production retry sleeps

**Issues Deferred:**

- backfill of missing phase `VERIFICATION.md` artifacts for phases 28, 29, 30, 31, and 31.1
- normalization of milestone closeout metadata across all generated GSD artifacts
- further reduction of the remaining non-E2E runtime outliers beyond the local 2-minute ceiling

**Technical Debt Incurred:**

- milestone was archived with accepted audit debt because implementation proof lives in summaries and targeted tests, not in the formal verification layer the audit workflow expects
- the full non-E2E suite still exceeded a strict 2-minute local ceiling even after the structural fixes in phase 31.1
