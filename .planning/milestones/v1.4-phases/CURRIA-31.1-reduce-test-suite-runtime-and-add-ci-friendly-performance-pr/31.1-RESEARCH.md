# Phase 31.1 Research: Reduce Test Suite Runtime and Add CI-Friendly Performance Proof  **Date:** 2026-04-15 **Phase:** 31.1-reduce-test-suite-runtime-and-add-ci-friendly-performance-pr  ## Goal  Turn the current non-E2E test runtime bottleneck into an explicit brownfield hardening slice: remove artificial latency, reduce the slowest suites, and leave the repo with a repeatable runtime-proof path before milestone closure.  ## Evidence Collected  ### The suite was paying browser-environment cost globally even for backend and library tests  - `vitest.config.ts` was configured with `environment: 'jsdom'` globally, so non-UI suites in `src/lib/**` and `src/app/api/**` paid DOM bootstrap cost they did not need. - A local fix already proved this is a real lever: switching the default environment to `node` and matching `jsdom` only for `*.test.tsx` made representative backend suites effectively free of environment startup cost. - Measured examples from the investigation:   - `src/app/api/agent/route.test.ts` completed in about `1.20s`   - `src/lib/agent/vacancy-analysis.test.ts` completed in about `748ms`   - those same classes of tests previously paid unnecessary jsdom setup  ### Real production backoff was leaking into retry and stream tests  - `src/lib/agent/agent-recovery.ts` used production retry timing that made tests wait for real backoff instead of behavior. - The profiled outliers showed a dramatic drop once delay became test-configurable:   - `src/lib/agent/streaming-loop.test.ts`: about `12.11s -> 86ms`   - `src/app/api/agent/route.sse.test.ts`: about `9.08s -> 95ms` - This is exactly the kind of artificial latency Phase 31.1 should formalize and finish: keep production semantics, but never force test suites to sleep when the assertion is about control flow rather than wall clock.  ### Several UI suites still spend meaningful time on simulated typing and repeated async polling  - The repo uses many Testing Library interactions, and the heaviest UI suites were still slow even after the environment split. - Local profiling during investigation already identified a pattern: repeated `userEvent.type(...)`, `user.clear(...)`, and long async interaction chains are inflating runtime when the assertion only depends on final field state or submit behavior. - The strongest measured wins so far came from replacing artificial typing with direct change or key events where typing realism was not the thing under test:   - `src/components/dashboard/chat-interface.route-stream.test.tsx`: about `5.76s -> 3.48s`   - `src/components/dashboard/chat-interface.test.tsx`: about `5.28s -> 819ms`   - `src/components/dashboard/job-applications-tracker.test.tsx`: about `6.04s -> 1.68s`   - `src/components/resume/user-data-page.test.tsx`: about `2.61s -> 1.34s`  ### The full non-E2E suite still exceeds the local timeout budget after the first wave of fixes  - Even after the biggest waste was removed, `npm test` still had to be cancelled after roughly `124s`. - That means the problem is no longer one hidden giant outlier. The remaining cost is distributed across additional medium-to-large suites, especially UI-heavy ones. - `src/components/resume/resume-builder.test.tsx` remains a known candidate at roughly `6s`, and other medium suites can still accumulate enough time to keep the full suite above the user's tolerance.  ### The next phase should lock in proof, not just ad-hoc speedups  - The repo now has working evidence of multiple performance wins, but that knowledge currently lives in local edits and conversation history rather than committed planning artifacts or explicit budget checks. - Phase 31.1 should therefore do three things in sequence:   1. capture and harden the baseline/profiling path   2. land the already-proven low-risk fixes plus the next outliers   3. add a durable runtime-proof command or CI-visible path so test runtime drift is observable  ## Recommended Plan Split  ### Wave 1  - `31.1-01`: Lock the runtime baseline and test-environment strategy so non-UI suites stop paying browser or backoff overhead by default, and add a repeatable profiling path for remaining outliers.  ### Wave 2  - `31.1-02`: Apply the proven low-risk optimizations to the highest-cost profiled suites, especially test-only retry delay control and direct input-event substitutions where typing realism is not under test.  ### Wave 3  - `31.1-03`: Finish the phase with runtime-proof artifacts: remeasure the full non-E2E suite, address or document the remaining largest outliers, and expose a durable budget/profiling command for future regressions.  ## Risks and Constraints  - Performance fixes must not weaken assertions or silently skip coverage just to hit a time target. - Some interactions really do depend on user-event semantics; where direct `fireEvent` changes alter behavior, the test should keep realistic interaction instead of forcing a speedup. - The suite still includes unknown medium-cost files, so the plan should preserve a profiling loop instead of assuming the currently known outliers are the whole story. - Milestone closure should stay paused until the runtime bottleneck is either materially reduced with proof or narrowed to an explicit residual risk with committed follow-up evidence.  ## Validation Architecture  ### Automated proof  1. Keep `npm run typecheck` green after every plan slice. 2. Re-run the profiled hot spots directly while optimizing them:    - `npm test -- src/lib/agent/streaming-loop.test.ts src/app/api/agent/route.sse.test.ts`    - `npm test -- src/components/dashboard/chat-interface.route-stream.test.tsx src/components/dashboard/chat-interface.test.tsx src/components/dashboard/job-applications-tracker.test.tsx src/components/resume/user-data-page.test.tsx` 3. Add or update one repeatable profiling or budget command that can be used locally and in CI-friendly checks for non-E2E runtime. 4. Re-measure the broader non-E2E suite with a bounded runtime check before phase closure.  ### Manual proof  1. Review any speed-up changes in UI tests and confirm they still assert the same user-visible behavior rather than lower-fidelity implementation details. 2. Confirm that retry-related tests still validate fallback control flow and not just mocked timing removal.  ### Success signal for Phase 31.1  Phase 31.1 is complete only when all of the following are true:  - non-E2E suites no longer pay obvious global environment or backoff costs that add no behavioral value - the slowest known suites are either materially faster or explicitly documented as deliberate residual cost - the repo exposes a repeatable runtime-proof path so future drift becomes visible before milestone closure or CI regression
