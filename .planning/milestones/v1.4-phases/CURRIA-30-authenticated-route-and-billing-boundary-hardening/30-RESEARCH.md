# Phase 30 Research: Authenticated Route and Billing Boundary Hardening  **Date:** 2026-04-15 **Phase:** 30-authenticated-route-and-billing-boundary-hardening  ## Goal  Remove weak trust assumptions from checkout, external return, webhook, and authenticated mutation flows so Phase 30 fails closed around billing and state-changing requests without breaking the existing brownfield product surface.  ## Evidence Collected  ### Checkout still trusts raw request origin for hosted billing callbacks  - `src/app/api/checkout/route.ts` currently reads `req.headers.get('origin') ?? 'http://localhost:3000'` and uses that value directly to build `successUrl` and `pricingUrl`. - This means the billing redirect contract is derived from caller-controlled metadata instead of canonical configuration, which conflicts directly with `SEC-01`. - The app already has canonical URL primitives in `src/lib/config/app-url.ts` (`getAppUrl()`, `getAppOrigin()`, `isCanonicalAppHost(...)`) and canonical-host redirect enforcement in `src/middleware.ts`, so Phase 30 should reuse those trust decisions instead of inventing a second host policy. - `src/app/api/checkout/route.test.ts` currently proves billing sequencing and failure handling, but it does not yet lock canonical-host derivation or rejection of unsafe return hosts.  ### There is no shared origin or CSRF guard for authenticated mutations today  - Repo-wide search found no shared CSRF or origin-validation helper for the authenticated API surface. - Multiple state-changing authenticated routes accept JSON and immediately mutate persistent state after auth:   - `src/app/api/profile/route.ts` (`PUT`)   - `src/app/api/profile/ats-enhancement/route.ts` (`POST`)   - `src/app/api/profile/smart-generation/route.ts` (`POST`)   - `src/app/api/session/[id]/manual-edit/route.ts` (`POST`)   - `src/app/api/session/[id]/gap-action/route.ts` (`POST`)   - `src/app/api/session/[id]/generate/route.ts` (`POST`)   - `src/app/api/session/[id]/targets/route.ts` (`POST`) - These routes rely on session auth only. For browser-initiated mutations, that leaves no explicit same-origin or CSRF contract, which is the main `SEC-02` gap. - The closest existing pattern is `src/app/api/e2e/auth/route.ts`, which already implements `isSameOriginRequest(...)`. That is a strong local precedent for a shared trust-boundary helper that can be generalized for production authenticated mutations.  ### Billing callback handling already has token auth and logging, but the boundary contract is fragmented  - `src/app/api/webhook/asaas/route.ts` already validates `asaas-access-token`, rate limits requests, parses payloads via zod, and emits structured logs for unauthorized, invalid, duplicate, ignored, processed, and failed webhook states. - This route is already stronger than the authenticated mutation routes, but it still belongs in Phase 30 because `SEC-03` asks for committed proof and clear trust-boundary behavior around billing callbacks and external flows as a set. - The main planning opportunity is to leave webhook token auth intact while adding clearer regression proof around accepted and rejected headers, and to keep checkout return-path trust aligned with the webhook and middleware trust model.  ### Canonical-host logic already exists, so the safest implementation path is consolidation rather than invention  - `src/middleware.ts` already redirects non-canonical `curria.com.br` hosts to `getAppUrl()` for normal page requests while exempting `/api/webhook/*`. - `src/lib/config/app-url.ts` already normalizes app URL configuration and exposes canonical origin/host helpers. - Phase 30 should therefore consolidate trust-boundary logic into small reusable helpers such as:   - canonical origin resolution for outbound billing URLs   - safe origin parsing and comparison for authenticated browser mutations   - optional route-level exemptions for webhook and E2E-only paths - This matches the milestone constraint to preserve the brownfield architecture while tightening sensitive seams.  ### Existing route suites make this phase a good fit for focused security regression proof  - There are already route tests for checkout and the main authenticated mutation surfaces:   - `src/app/api/checkout/route.test.ts`   - `src/app/api/profile/ats-enhancement/route.test.ts`   - `src/app/api/profile/smart-generation/route.test.ts`   - `src/app/api/session/[id]/manual-edit/route.test.ts`   - `src/app/api/session/[id]/gap-action/route.test.ts`   - `src/app/api/session/[id]/targets/route.test.ts`   - `src/app/api/session/[id]/generate/route.test.ts`   - `src/app/api/webhook/asaas/route.test.ts` - That means Phase 30 does not need a new testing framework. It should add focused negative-path coverage for invalid origin or CSRF context and for canonical checkout return derivation.  ### The cleanest split is host and return-path hardening first, then mutation enforcement and callback proof  - Plan `30-01` should focus on `SEC-01` first:   - move checkout success/cancel URL derivation behind canonical configuration   - reject unsafe return targets or malformed external redirect inputs   - add regression proof around those trust decisions - Plan `30-02` should then cover `SEC-02` and `SEC-03` together:   - introduce a shared authenticated-mutation origin or CSRF guard   - apply it to high-value state-changing authenticated routes   - extend webhook and billing tests so the overall trust boundary is explicit and operator-visible - This ordering keeps the billing redirect hole from lingering while avoiding one oversized security patch touching every route at once.  ## Recommended Plan Split  ### Wave 1  - `30-01`: Harden canonical-host and return-path validation across checkout and any external redirect handoffs that currently trust request metadata.  ### Wave 2  - `30-02`: Add explicit origin or CSRF enforcement to sensitive authenticated mutations and extend billing callback regression proof around accepted and rejected trust signals.  ## Risks and Constraints  - The brownfield product surface should not change unnecessarily; the phase should harden trust boundaries without redesigning the checkout or workspace UX. - Canonical-host enforcement must preserve localhost and preview-safe behavior where the current app intentionally allows it. - Authenticated mutation guards should fail closed for missing or invalid browser trust signals, but they must not break server-to-server webhook flows or E2E-only endpoints that already have a separate contract. - Security tightening should prefer one shared helper over route-by-route ad hoc checks so future phases can reuse the same trust model.  ## Validation Architecture  ### Automated proof  1. Keep `npm run typecheck` green after each plan slice. 2. Use `npm test -- src/app/api/checkout/route.test.ts src/app/api/webhook/asaas/route.test.ts` as the primary regression suite for canonical-host and billing callback trust changes. 3. Use the authenticated mutation suites as the second plan's targeted regression net:    - `src/app/api/profile/ats-enhancement/route.test.ts`    - `src/app/api/profile/smart-generation/route.test.ts`    - `src/app/api/session/[id]/manual-edit/route.test.ts`    - `src/app/api/session/[id]/gap-action/route.test.ts`    - `src/app/api/session/[id]/targets/route.test.ts`    - `src/app/api/session/[id]/generate/route.test.ts` 4. Add direct unit coverage for any new trust-boundary helper introduced in this phase.  ### Manual proof  1. Start a real checkout from the browser and confirm the hosted return path resolves back to the canonical app URL rather than whatever `Origin` the request happened to send. 2. Attempt one authenticated browser mutation from a mismatched or missing origin context in dev tooling and confirm the route rejects it cleanly with a fail-closed response.  ### Success signal for Phase 30  Phase 30 is complete only when all of the following are true:  - checkout and external return flows derive trusted URLs from canonical configuration instead of raw request metadata - sensitive authenticated mutations require explicit origin or CSRF trust signals and reject invalid callers - billing callback and checkout boundary behavior have committed regression proof plus structured logs around allowed and rejected trust paths
