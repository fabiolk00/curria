# Phase 28 Research: Agent Input and Setup Service Extraction  **Date:** 2026-04-15 **Phase:** 28-agent-input-and-setup-service-extraction  ## Goal  Pull the front half of the agent runtime into smaller services so message preparation, vacancy detection, and pre-loop setup are easier to reason about, test, and evolve without changing user-visible behavior by default.  ## Evidence Collected  ### The current split between `route.ts` and `agent-loop.ts` already exposes the seams this phase should formalize  - `src/app/api/agent/route.ts` owns request validation, sanitization, target-job detection, workflow-mode prediction, session bootstrap, SSE setup, and `runPreLoopSetup(...)`. - `src/lib/agent/agent-loop.ts` still owns append or read message history, deterministic fast paths, repeated vacancy detection, prompt construction, and the full loop orchestration. - The route currently calls `runAgentLoop(...)` only after doing front-loaded mutation and setup work, which means the safest phase-28 extraction is not a rewrite of behavior but a formalization of the helper boundaries that already exist.  ### Vacancy and target-job detection are duplicated across the route and the loop  - `route.ts` contains `detectTargetJobDescription(...)` and `normalizeForJobDescriptionDetection(...)`. - `agent-intents.ts` contains `looksLikeJobDescription(...)`. - `agent-loop.ts` repeatedly recomputes `latestMessageLooksLikeVacancy` and `hasTargetJobContext` in multiple branches. - This duplication is a direct planning signal: Phase 28 should extract one shared message or vacancy analysis service and move both route and loop to it before later decomposition phases touch retry, streaming, or persistence.  ### The highest-value phase-28 seams are message preparation, vacancy analysis, and pre-loop setup contracts  - In `route.ts`, message preparation currently includes `sanitizeUserInput(...)`, scraped-job normalization, and auto-detection of target-job content. - In `route.ts`, `runPreLoopSetup(...)` is already a distinct helper with new-session and existing-session branches, which makes it the right extraction candidate instead of introducing a brand-new abstraction first. - In `agent-loop.ts`, the loop entry still begins with side effects (`appendMessage`, `getMessages`, `buildSystemPrompt`, `trimMessages`) before branching into deterministic behaviors. - The safest extraction order is therefore:   1. Shared message input or vacancy analysis utilities used by both route and loop.   2. A narrower pre-loop setup service that owns the existing helper behavior and route-facing contract.   3. Focused tests around the extracted entry flow before later phases move deeper runtime behavior.  ### Route and loop tests already provide the regression harness this phase needs  - `src/app/api/agent/route.test.ts` and `src/app/api/agent/route.sse.test.ts` already assert route-to-loop integration and SSE behavior. - `src/lib/agent/streaming-loop.test.ts` already exercises `runAgentLoop(...)` with mocked history, prompt building, tool dispatch, and streaming. - That means Phase 28 can add coverage at the seam level instead of creating a new framework:   - route-level tests for sanitized input and pre-loop delegation   - unit tests for shared vacancy or message preparation utilities   - loop-entry regression tests proving behavior did not change when setup responsibilities move  ### `cvState` and `agentState` boundaries must stay explicit during extraction  - The project instructions in `AGENTS.md` explicitly require `cvState` to remain canonical truth and `agentState` to remain operational context only. - `route.ts` currently decides workflow mode from both `cvState` and `agentState`, while `agent-loop.ts` uses saved target-job data and phase metadata for deterministic branches. - Any extracted setup service must preserve those boundaries rather than smuggling durable resume truth into new runtime-local helpers. - This means the plan should prefer pure or mostly pure helpers for message analysis and route-to-loop preparation, with mutation concentrated in existing persistence seams.  ### The phase should not absorb recovery, streaming, or persistence extraction yet  - `agent-loop.ts` still mixes deterministic handling, tool execution, streaming retries, persistence, and fallback logic deep into the loop body. - However, roadmap Phase 29 already owns retry, recovery, streaming, and persistence decomposition. - Phase 28 should therefore stop at front-half extraction boundaries and regression coverage, so the milestone stays staged and checker-friendly.  ## Recommended Plan Split  ### Wave 1  - `28-01`: Extract route-level message preparation and shared vacancy-analysis logic into dedicated agent services used by `route.ts` and loop-adjacent callers. - `28-02`: Narrow `runPreLoopSetup(...)` into an explicit pre-loop setup service with a stable route-facing contract and clearer read or write responsibilities.  These can run in parallel only if the shared vacancy-analysis module lands without changing the pre-loop setup signature. If coupling is higher than expected, treat them as sequential within the same wave.  ### Wave 2  - `28-03`: Add regression tests around the extracted setup path, including route-to-loop behavior, sanitized input handling, and target-job detection preservation.  This belongs after extraction so the tests lock the final seam instead of the intermediate helper arrangement.  ## Risks and Constraints  - Phase 28 must not change the brownfield product surface for `/api/agent`, especially session bootstrap timing and SSE startup expectations. - The route already emits `sessionCreated` before slow setup for new sessions; extracted pre-loop services must preserve that contract. - Vacancy detection heuristics are user-visible because they influence target-job adaptation and deterministic bootstrap behavior; changing thresholds in this phase would couple architecture work to behavior changes. - The large `agent-loop.ts` test suite is a strength, but it also means careless signature churn can create noisy failures; plan slices should keep public contracts narrow.  ## Validation Architecture  ### Automated proof  1. Keep `npm run typecheck` green after each extraction slice. 2. Run focused Vitest coverage for `src/app/api/agent/route.test.ts`, `src/app/api/agent/route.sse.test.ts`, and `src/lib/agent/streaming-loop.test.ts`. 3. Add or extend tests for the extracted message-preparation or vacancy-analysis helpers so route and loop callers share one asserted contract. 4. Re-run the route-model or SSE suite if setup signatures change where `runAgentLoop(...)` is invoked.  ### Manual proof  1. Exercise one new-session request and one existing-session request to confirm the route still emits `sessionCreated` early and streams normally. 2. Replay one pasted vacancy message and one resume-only message to confirm target-job detection and workflow-mode behavior did not drift.  ### Success signal for Phase 28  Phase 28 can be considered complete only when all of the following are true:  - route-level message preparation and vacancy detection are no longer duplicated across multiple ad hoc helpers - pre-loop setup has an explicit service boundary instead of living as route-local orchestration glue - the route-to-loop handoff still preserves current behavior for session creation, workflow prediction, and initial streaming - committed tests protect the extracted entry flow so Phase 29 can safely decompose deeper runtime behavior
