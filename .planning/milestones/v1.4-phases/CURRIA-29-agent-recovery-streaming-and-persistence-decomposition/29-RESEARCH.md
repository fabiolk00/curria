# Phase 29 Research: Agent Recovery, Streaming, and Persistence Decomposition  **Date:** 2026-04-15 **Phase:** 29-agent-recovery-streaming-and-persistence-decomposition  ## Goal  Split the back half of the brownfield agent runtime into clearer retry, recovery, streaming, and persistence services so Phase 28's cleaner route-to-loop seam can hand off into smaller, testable runtime boundaries without changing user-visible behavior by default.  ## Evidence Collected  ### `agent-loop.ts` still concentrates too many back-half responsibilities in one execution path  - `src/lib/agent/agent-loop.ts` mixes history persistence (`appendMessage`, `getMessages`), streaming request orchestration, token or empty-text recovery, deterministic fast paths, tool execution, patch emission, usage tracking, and final fallback behavior. - The route now hands off through a clearer `AgentLoopParams` contract after Phase 28, which means the next highest-value work is inside the loop itself rather than at the route boundary. - The current file already exposes natural seams by function cluster:   - `streamAssistantTurn(...)` for streaming transport and first-token logging   - `recoverTruncatedTurn(...)`, `recoverZeroTextTurn(...)`, and `recoverConciseTurn(...)` for recovery policy   - repeated `appendMessage(...)`, `getMessages(...)`, `applyToolPatchWithVersion(...)`, and done-chunk logging for persistence or completion handling  ### Retry and degraded-recovery policy is duplicated across multiple late-turn paths  - The loop has one recovery branch for `finishReason === 'length'`, another for empty assistant text, and a later multi-attempt fallback block when `assistantResponded` is still false. - These paths all rely on the same primitives:   - `streamAssistantTurn(...)`   - `buildRecoveryUserPrompt(...)`   - `RECOVERY_SYSTEM_PROMPT`   - usage tracking and fallback-kind logging - The late fallback block also adds exponential backoff logic inline, which makes policy changes risky because recovery behavior is split across multiple distant sections of the file. - Phase 29 should therefore extract recovery orchestration first, before moving streaming or persistence, so the remaining loop logic can depend on one recovery contract instead of open-coded branches.  ### Streaming transport and persistence handoffs are entangled in the main loop body  - `streamAssistantTurn(...)` owns request assembly, model selection, OpenAI stream consumption, text chunk emission, tool-call accumulation, and first-token observability. - The main `runAgentLoop(...)` body then immediately couples stream results to message persistence, tool-history mutation, patch yielding, and final `done` events. - There is no single explicit service boundary today for:   - loading and trimming history   - appending assistant output after a turn   - applying persisted tool patches and marking prompt dirtiness   - finishing a turn with logs and a `done` chunk - This is the right second extraction slice after recovery, because it narrows the loop into orchestration over smaller runtime services instead of continuing to own both transport and state mutation details.  ### Phase 28 tests already provide the front-half safety net, and Phase 29 should extend the back-half proof rather than invent a new framework  - `src/lib/agent/streaming-loop.test.ts` already covers large portions of `runAgentLoop(...)`, including truncation handling, deterministic fallbacks, tool execution, and transcript persistence. - `src/app/api/agent/route.test.ts` and `src/app/api/agent/route.sse.test.ts` now lock the route-to-loop handoff created in Phase 28. - The highest-leverage testing move in Phase 29 is:   1. unit or seam-level tests for extracted recovery and persistence helpers   2. targeted updates to `streaming-loop.test.ts` that prove the new service boundaries preserve observable behavior   3. route-level smoke proof only where handoff semantics could drift  ### Canonical `cvState` and operational `agentState` boundaries remain the non-negotiable invariant  - Project constraints still require `cvState` to remain product truth and `agentState` to remain runtime context only. - The risky parts of the loop are where tool patches are applied and where deterministic generation or rewrite flows persist assistant output, phase changes, or targeting state. - Any extracted persistence service must keep using the current durable seams (`appendMessage`, `applyToolPatchWithVersion`, `trackApiUsage`) rather than introducing a second write path for resume truth.  ### The cleanest phase split is recovery first, then streaming plus persistence, then regression proof  - Recovery extraction should land first because it mostly reorganizes policy around existing helpers and reduces the branching complexity of the remaining loop. - Streaming and persistence extraction should follow once recovery policy has been narrowed, because both plans will reshape `agent-loop.ts` materially and should not compete in parallel. - The testing plan should close the phase after those extractions so the new seams are protected before Phase 31 depends on runtime stability.  ## Recommended Plan Split  ### Wave 1  - `29-01`: Extract retry and degraded-recovery policy into dedicated runtime helpers or services used by `runAgentLoop(...)`.  ### Wave 2  - `29-02`: Separate streaming transport, transcript persistence, and patch-application handoffs from the main loop body while preserving logs and done-chunk semantics.  ### Wave 3  - `29-03`: Add direct regression coverage for recovery, streaming, and persistence seams, plus targeted loop and route proof for the preserved handoff contract.  ## Risks and Constraints  - Phase 29 must not regress first-token streaming, deterministic rewrite or generation fast paths, or transcript persistence ordering. - Recovery extraction is behavior-sensitive because it affects truncation, empty-text fallback, and operator-visible logging. - Streaming and persistence extraction should not create a second notion of canonical state; they must continue to route durable changes through the existing session persistence APIs. - The file is large enough that parallel plan ownership would create merge risk, so the plans should stay sequential.  ## Validation Architecture  ### Automated proof  1. Keep `npm run typecheck` green after every plan slice. 2. Use `npm test -- src/lib/agent/streaming-loop.test.ts` as the primary fast loop suite for recovery, streaming, and persistence changes. 3. Re-run `npm test -- src/app/api/agent/route.test.ts src/app/api/agent/route.sse.test.ts` whenever the route-to-loop handoff contract or observable stream startup semantics could drift. 4. Add direct tests for any new extracted recovery or persistence module introduced in this phase.  ### Manual proof  1. Exercise one existing-session chat turn that returns normal assistant text and confirm transcript persistence still matches the visible stream. 2. Exercise one degraded or empty-response scenario and confirm the user still receives the same class of fallback instead of a silent stream end.  ### Success signal for Phase 29  Phase 29 is complete only when all of the following are true:  - retry and degraded-recovery policy no longer live as scattered branches inside one oversized loop path - streaming transport and persistence handoffs are explicit instead of embedded line-by-line throughout `runAgentLoop(...)` - the extracted runtime seams preserve canonical `cvState`, operational `agentState`, and deterministic generation persistence contracts - committed tests protect the new back-half seams so later stability and release work can depend on them safely
